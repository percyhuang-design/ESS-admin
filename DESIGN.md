# ESS × HR Admin 升級設計

> 本次升級目標:在既有「員工自助平台 (ESS)」上,新增 **HR Admin 審核功能**,
> 讓各業務單位的 HR 審核員工提出的資料更新申請單,並支援結案查詢。

---

## 1. 整體概念:申請單是兩端的橋

```
員工端 (ESS)                          HR Admin 端
─────────────                        ─────────────
自助項目頁修改資料                     審核中心(收件匣)
   │ 送出申請                            │ 只見自己負責分類;管理員全視野
   ▼                                    ▼
ChangeRequest(待審核) ───────────▶  審核:同意 / 拒絕(理由必填)
   ▲                                    │
   │ 通知 + 我的申請紀錄  ◀──────────────┘
   │   已核准 → 寫回主檔
   └── 已退回 → 顯示理由,可修改後重送(resubmitOf 鏈結原單)
```

兩端共用同一個核心實體 **ChangeRequest(申請單)**,不是兩套系統:
員工提交即進入負責 HR 的收件匣;HR 審核結果即時回到員工的「我的申請紀錄」。

## 2. 角色與權限模型

| 動作 | 員工 | HR(負責分類) | 平台管理員 |
|---|---|---|---|
| 提交/重送申請 | ✅ 自己的 | — | — |
| 查看申請單 | 自己的 | 負責分類內全部 | 全部(可依分類篩選) |
| 審核(同意/拒絕) | — | ✅ 負責分類 | ⚠️ 待確認(見下) |
| 結案查詢 | 自己的 | 負責分類 | 全部 |

- **分類負責制**:每個自助項目(ServiceItem)隸屬一個**業務分類(BizCategory)**,
  申請單提交時帶入該分類;HR 帳號設定 `categories[]` 即「負責哪些分類」。
  權限過濾的單位是「分類」而非「項目」,新增自助項目時只要掛上分類,審核權限自動生效。
- **管理員全視野**:admin 不設 categories,直接可見全部分類,並以分類 Tabs/篩選器檢視。
- **⚠️ 待確認決策**:管理員是否可「代審」?目前 `canReview()` 預設**僅負責 HR 可審、
  admin 只能檢視督導**(權責清楚、稽核單純);若需代審,放寬 `canReview` 一行即可。
- 正式版角色來源:SSO 登入 + HR 系統的單位/職務對應表(本骨架以 Demo 身分切換模擬)。

### 業務分類 × 自助項目對應(可依實際組織調整)

| 業務分類 | 負責單位(例) | 自助項目 |
|---|---|---|
| 人事資料 personal | 人事資料組 | 基本資料、聯絡資訊、緊急聯絡人 |
| 薪酬作業 payroll | 薪酬組 | 薪轉帳戶 |
| 保險福利 insurance | 保險福利組 | 眷屬資料 |
| 學習發展 development | 學習發展組 | 學歷與證照 |

## 3. 申請單資料模型與狀態機

```
pending(待審核) ──同意──▶ approved(已核准)  ┐
      │                                     ├─ 即「結案」,進入結案查詢
      └────拒絕(理由必填)──▶ rejected(已退回) ┘
                                  │
                                  └─ 員工修改後重送 = 產生新單,resubmitOf 鏈結原單
```

`ChangeRequest` 關鍵欄位(見 `src/data/mock.ts`):

- `changes: FieldChange[]` — **異動前後對照**是審核的核心資訊,HR 一眼看出改了什麼。
- `category` — 提交時由項目帶入,凍結在單上(權限過濾與篩選依據,項目改組不影響歷史單)。
- `rejectReason` — `rejected` 時必填;直接呈現給員工。
- `reviewerName / reviewedAt` — 審核紀錄,結案查詢與稽核用。
- `resubmitOf` — 重送鏈結,審核時可回看前次被退回的原因(設計保留,後續實作)。
- 設計原則:**申請單 append-only**——退回後不改原單,重送是新單;歷史完整可稽核。

## 4. HR Admin 功能設計

### 4.1 審核中心(收件匣)— `pages/admin/ReviewCenter.tsx`
- 進來只看見「自己負責分類」的**待審**申請單;admin 看全部 + 分類篩選 Tabs。
- 列表欄位:單號/員工/部門/項目/分類/**異動內容**(`ChangeLines`,多欄位堆疊呈現 before→after)/送出時間/操作。
- 操作欄=**三個 icon**:👁 看申請單 / ✓ 同意 / ✗ 退回(Tooltip);管理員唯讀只有「看」。同意/退回可**直接開確認 Modal**,免先開審核 Modal。
- 審核彈窗(`RequestReviewModal.tsx`):置中 Modal,**列出該項目完整資料(含未異動欄位)供整體評估**——本次異動列以琥珀底色高亮 + 「異動」tag + before→after,未異動列灰階;**只負責顯示 + 發出意圖**。
- 資料模型:`ChangeRequest.allFields`(整份欄位,`after` 標記異動)為單一來源,`changesOf()` 衍生 delta 供列表/批次/確認摘要共用。
  按「同意/拒絕」→ **關閉審核 Modal、改開獨立確認 Modal**(`ConfirmActionModal`,**循序、不堆疊**,避免 modal-on-modal 與按鈕原地變身)——
  核准=確認訊息(薪轉帳戶等金流加警示)、退回=理由必填(空白 disabled);確認 Modal **取消則返回審核 Modal**。
  讀取/已結案為唯讀(無 footer)。互動演進:Drawer → push-panel → Modal+就地內嵌 → **Modal+循序確認 Modal(定案)**。
  注意:兩個 Modal 交換時有約 0.3s 過場(一瞬只剩遮罩),屬循序 Modal 固有現象。
- 側欄「審核中心」badge 顯示可見範圍內的待審數(骨架已接通)。

### 4.2 全部申請紀錄 — `pages/admin/RequestRecords.tsx`(前身「結案查詢」)
- 範圍:**含待審核在內的所有申請單**(不再只看結案);可見分類與審核中心一致(RBAC)。
- **主要條件(常駐)**:申請單號 / 申請人(姓名·工號) / 審核狀態(全部/待審核/已核准/已退回)。
- **進階選項用 Popover**(`進階選項` 鈕;有套用時鈕上顯示 •):分類 / 審核人(下拉=資料中出現過的審核人) / 申請區間 / 審核區間 + 重置進階/套用。
  - Popover **完全受控**(`trigger={[]}` + 按鈕 toggle):因內含 RangePicker 日曆是 portal 到 body 的浮層,若用預設外部點擊關閉會「點日曆=點外面」而誤關;受控後只由按鈕開關。
- **區間=`components/DateRangeField`**:**Radio 快捷**(不指定/近30天/近半年/自訂),比藏在日曆下拉的 presets 直觀。針對 admin 歷史查詢只留「近況(月)/回顧(半年)」兩尺度,7天/90天屬當前工作節奏(歸審核中心)故略;跨年度查詢用「自訂」。選「自訂」才展開 `RangePicker`(預設不顯示),`mode` state + `deriveMode()` 反推、外部清除同步。需要可再加「近一年」。
- **初始空狀態**:`searched` 旗標,進頁面**先按過一次「搜尋」下方才出資料**;未搜尋顯示引導 Empty(含搜尋鈕)、標題不顯示筆數。
- **互動模式=按「搜尋」才套用**:所有條件(主要+進階)為 draft,按「搜尋/套用」才寫入 applied 並過濾(輸入框 Enter=搜尋);非即時。`清除` 重置全部並**回到初始空狀態**(需再次搜尋)。
- 明細 Modal 唯讀(動作仍走審核中心)。匯出(後續)。

### 4.3 批次審核 — 審核中心的勾選 + 批次動作
審核本質是逐筆把關,批次化要避免變成橡皮圖章,故核准/退回分開設計:
- **批次核准**:適合同質低風險量(例:搬遷後一票人改地址)。確認 Modal **列出選取清單並內嵌異動內容**(每筆「欄位:原值→新值」),讓審核者真的看得到在核准什麼、不是閉眼全過;含 **payroll(薪轉帳戶,金流盜用風險高)** 時跳警示提醒逐筆確認。
- **批次退回**:保留紅線「**拒絕理由必填**」,但只能套**同一理由**到多筆,故 UI 明示「此理由將套用到全部 N 筆」。僅在理由真的共通時適用(例:受理期截止、統一補件);否則請逐筆退以給個別理由。
- **權限**:只有「該分類負責 HR」出現勾選框;**管理員唯讀,無勾選**(與單筆只給「檢視」一致)。`getCheckboxProps` 再擋一層非可審列。換篩選/換身分自動清空選取。
- **狀態**:`useRequests.approveMany/rejectMany` 一次更新,僅作用於仍為 `pending` 的單(避免重覆結案)。

## 5. 與 ESS 的整合方式(本設計的核心主張)

**同一平台、同一登入、側欄依角色長出 HR 區塊**,而非獨立的 Admin 系統:

1. **側欄 IA**:「員工自助」群組人人可見(HR 也是員工);
   「HR 審核管理」群組僅 `hr` / `admin` 出現(審核中心、結案查詢)。
2. **共用資料模型**:員工端送出申請與 HR 端審核操作同一張 `ChangeRequest`,
   不需同步、不會有兩邊狀態不一致的問題(單一真實來源)。
3. **閉環回饋**:審核結果直接反映在員工端「我的首頁」待辦與「我的申請紀錄」;
   退回理由原文呈現,員工可一鍵帶入原內容修改重送。
4. **權限在資料層把關**:前端選單只是 UX,真正的隔離靠後端以
   「分類負責制」過濾查詢(正式版即 API / RLS 層的 where 條件),
   HR 換組只改負責分類設定,不動程式。
